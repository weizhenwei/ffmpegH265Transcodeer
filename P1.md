# 分布式 H.265 短视频转码系统需求说明书（P1）

## 1. 文档信息
- 文档版本：P1.0
- 文档类型：产品需求说明书（PRD）
- 适用范围：第一阶段（P1）可落地实现版本

## 2. 项目背景与目标
### 2.1 背景
当前短视频内容规模持续增长，现有单机转码方式在处理时效、吞吐能力和成本效率方面存在瓶颈。需要构建可横向扩展的分布式转码系统，统一完成视频格式检测、H.265 转码、输出管理与任务调度。

### 2.2 总体目标
- 提供高效的短视频转码服务，支持 H.265 编码。
- 支持多机并行转码，提升处理吞吐量。
- 支持本地、网络和云端共享存储的数据输入输出。
- 支持命令行触发、定时触发两种运行方式。
- 通过 Python 调用 FFmpeg 二进制程序完成转码。

### 2.3 P1 范围边界
**P1 包含：**
- Master-Slave 调度模型。
- 目录级批量转码任务。
- MP4、M3U8 输入，统一转码为 H.265（可封装为 MP4 或 TS/HLS）。
- Docker 化部署。
- 基础任务状态管理与日志记录。

**P1 不包含：**
- 复杂负载均衡策略（LB 可暂不考虑）。
- 多租户权限系统。
- 图形化 Web 控制台（可先以 CLI 为主）。

## 3. 术语定义
- Master：调度节点，负责扫描输入目录、切分任务、分发与状态汇总。
- Slave（Worker）：执行节点，负责拉取任务并调用 FFmpeg 执行转码。
- 任务（Job）：一次批处理请求，包含输入目录、输出目录、转码参数等。
- 子任务（Task）：Job 内针对单个输入文件的一次转码执行单元。
- 共享存储：Master 与 Worker 都可访问的统一存储空间（本地挂载/NAS/对象存储网关）。

## 4. 角色与使用场景
### 4.1 目标用户
- 运维工程师：部署与维护集群、配置调度策略。
- 媒体处理工程师：提交转码任务、查看结果与失败原因。

### 4.2 核心场景
- 手动触发：用户通过命令行提交目录转码任务。
- 定时触发：系统按 Cron 规则自动扫描目录并提交任务。
- 多机并行：Master 分配子任务给多个 Worker 并发处理。
- 失败重试：单文件转码失败后按策略重试并记录错误。

## 5. 功能需求
### 5.1 任务触发与管理
1. 系统应支持命令行手动触发任务。
2. 系统应支持定时触发任务（Cron 表达式）。
3. 每个任务至少包含以下参数：
   - 输入目录路径
   - 输出目录路径
   - 输出文件后缀策略
   - 转码参数模板或自定义参数
   - 最大并发数
   - 重试次数
4. 系统应生成唯一 Job ID，并支持按 Job ID 查询状态。
5. 任务状态至少包含：`PENDING`、`RUNNING`、`PARTIAL_SUCCESS`、`SUCCESS`、`FAILED`。

### 5.2 输入扫描与格式检测
1. Master 应扫描输入目录下全部待处理文件。
2. 系统应过滤不支持格式并记录原因。
3. 系统应调用 FFmpeg/ffprobe 检测媒体信息（容器、时长、分辨率、编码格式）。
4. 支持的输入格式（P1）：
   - MP4
   - M3U8（点播切片索引）
5. 对无法读取或损坏文件应标记失败，不影响其他文件处理。

### 5.3 转码执行
1. Worker 必须通过 Python 调用 FFmpeg 二进制命令执行转码。
2. 转码模式为一对一：一个输入文件对应一个输出文件（或输出目录结构）。
3. 默认编码器为 H.265（libx265 或指定硬件编码器）。
4. 支持参数可配置，至少包括：
   - 视频编码器（默认 libx265）
   - 码率/CRF
   - 帧率
   - 分辨率缩放策略
   - GOP
   - 音频编码参数（可透传或转码）
5. 输出命名规则：在原文件名基础上追加后缀（如 `_h265`）。
6. 输出目录结构应可配置为：
   - 平铺输出
   - 保留输入目录层级

### 5.4 调度与并行处理
1. 系统采用 Master-Slave 架构。
2. Master 应支持将子任务均匀分发至多个 Worker。
3. Worker 数量可横向扩展，支持动态增减（P1 允许重启后生效）。
4. 同一 Worker 可配置并发执行数。
5. 调度策略（P1）采用基础轮询或队列拉取策略，保证实现简单稳定。

### 5.5 存储支持
1. 系统应支持本地存储路径读写。
2. 系统应支持网络共享存储（如 NFS/SMB）读写。
3. 系统应支持云存储挂载路径或对象存储网关路径读写。
4. Master 与 Worker 必须能访问同一输入输出路径命名空间。

### 5.6 日志、状态与结果
1. 系统应记录 Job 级与 Task 级日志。
2. 每个子任务应输出：
   - 输入文件路径
   - 输出文件路径
   - 开始/结束时间
   - 执行耗时
   - 执行结果
   - 失败原因（stderr 摘要）
3. 系统应生成任务汇总结果：
   - 总文件数、成功数、失败数、跳过数
   - 总耗时
   - 吞吐指标（文件/分钟）

## 6. 非功能需求
### 6.1 性能
- 支持多机并行，整体吞吐应随 Worker 数量增加而线性提升（允许合理损耗）。
- 单个 Worker 并发能力可配置，默认建议 1~4 并发（由机器资源决定）。

### 6.2 可靠性
- 单个子任务失败不应导致整个 Job 中断。
- 支持失败重试机制（默认 1~3 次可配）。
- Master 异常重启后应可恢复未完成任务状态（P1 可通过状态文件/数据库实现）。

### 6.3 可运维性
- 所有服务支持 Docker 部署。
- 配置项集中管理（配置文件 + CLI 参数覆盖）。
- 日志可按日期滚动，便于排查问题。

### 6.4 安全性
- 不在日志中输出敏感凭证（如云存储访问密钥）。
- 外部命令调用参数需要做转义与校验，防止命令注入。

## 7. 系统架构要求
### 7.1 逻辑架构
- 调度层：Master（扫描、拆分、分发、汇总）。
- 执行层：Workers（拉取任务、执行 FFmpeg、回传状态）。
- 存储层：共享输入输出存储。
- 元数据层：任务状态存储（可选 SQLite/PostgreSQL/Redis，P1 可先用轻量方案）。

### 7.2 通信要求
- Master 与 Worker 之间应通过可靠队列或 RPC 机制通信。
- 需支持心跳或存活检测，识别失联 Worker。

## 8. 配置需求
系统应支持以下配置类别：
- 基础配置：服务端口、节点角色、节点 ID。
- 调度配置：最大并发任务数、重试次数、超时时间。
- 转码配置：编码器、CRF/码率、封装格式、输出后缀。
- 存储配置：输入输出根路径、路径映射规则。
- 触发配置：定时规则、手动任务默认参数。

## 9. 命令行接口需求（CLI）
### 9.1 基本要求
- 提供统一 CLI 入口（如 `python main.py ...`）。
- 提供 `--help` 查看参数说明。

### 9.2 关键命令（示例规范）
- 提交任务：指定输入目录与输出目录。
- 查询任务：按 Job ID 查看状态与统计。
- 重试失败：对失败子任务重新执行。
- 停止任务：可选，按 Job ID 停止未开始子任务（P1 可选实现）。

## 10. 错误处理需求
- 输入目录不存在：任务创建失败并返回明确错误。
- 输出目录不可写：任务创建失败或子任务失败并记录原因。
- FFmpeg 不可用：服务启动时自检失败并阻止运行。
- 文件解码失败：标记子任务失败并继续其他任务。
- Worker 宕机：其未完成任务应回收并重新分发。

## 11. 监控与告警需求（P1 基础版）
- 至少暴露以下指标：
  - 当前排队任务数
  - 当前执行任务数
  - Job 成功率
  - 平均转码耗时
- 支持日志级别配置（INFO/WARN/ERROR）。
- 关键失败场景应输出可检索的错误码或错误关键字。

## 12. 验收标准
### 12.1 功能验收
- 能通过 CLI 成功提交目录转码任务。
- 能识别 MP4、M3U8 并完成 H.265 转码输出。
- 输出文件命名符合后缀规则并写入指定目录。
- 多 Worker 条件下可并行处理并完成汇总统计。

### 12.2 稳定性验收
- 在包含失败文件的批任务中，其他文件可继续处理。
- Worker 异常退出后，未完成任务可被重新调度。

### 12.3 部署验收
- Master 与 Worker 均可通过 Docker 容器启动。
- 使用共享存储后，跨机器读写输入输出路径正常。

## 13. 里程碑建议（P1 内部执行）
- M1：单机 CLI 转码闭环（扫描、检测、转码、输出）。
- M2：Master-Slave 基础调度可用。
- M3：多机并行 + Docker 部署可用。
- M4：日志、统计、重试、基础监控补齐并完成验收。

## 14. 后续演进方向（P2+）
- 引入更精细的负载均衡与优先级调度。
- 增加 Web 控制台与权限管理。
- 支持更多输入输出协议与格式。
- 引入自动扩缩容与成本优化策略。
