# 分布式 H.265 转码系统技术方案（T1）

## 1. 文档信息
- 文档版本：T1.0
- 对应需求：P1（`P1.md`）
- 目标：将 P1 需求落地为可实现、可部署、可运维的技术设计方案

## 2. 设计目标与原则
### 2.1 目标
- 支持目录级批量转码（MP4/M3U8 输入，H.265 输出）。
- 支持 Master-Slave 多机并行调度与执行。
- 通过 Python 调用 FFmpeg/ffprobe 二进制程序。
- 支持手动触发与定时触发。
- 支持失败重试、状态追踪、日志与基础监控。

### 2.2 设计原则
- 简单优先：P1 选择实现成本低、可维护的组件组合。
- 稳定优先：单任务失败不影响批任务整体推进。
- 可扩展优先：接口、状态机、配置结构为 P2 扩展预留。
- 可观测优先：统一结构化日志与指标，支持快速定位问题。

## 3. 技术选型
## 3.1 语言与运行时
- Python 3.11+
- 关键依赖建议：
  - `typer`：CLI 入口与参数管理
  - `pydantic`：配置与请求参数校验
  - `sqlalchemy` + `sqlite/postgresql`：任务元数据存储
  - `redis`：任务队列、心跳与轻量锁
  - `apscheduler`：定时任务触发
  - `structlog` 或标准 logging JSON formatter：结构化日志
  - `prometheus_client`：指标暴露

## 3.2 基础设施
- 队列：Redis Streams（P1 推荐）
  - 理由：部署简单、性能可接受、支持消费组与 ACK。
- 元数据库：
  - 开发/单机：SQLite
  - 生产：PostgreSQL
- 存储：共享目录（本地挂载/NFS/SMB/对象存储网关挂载）
- 容器化：Docker + docker-compose（P1）

## 4. 总体架构
## 4.1 组件划分
- `master`：扫描目录、创建 Job/Task、投递队列、汇总状态、触发重试。
- `worker`：消费任务、执行探测与转码、上报结果、发送心跳。
- `scheduler`（可内嵌 master）：管理 Cron 触发策略。
- `state-store`：保存 Job/Task、节点状态、重试记录。
- `queue`：解耦任务生产与消费。
- `shared-storage`：输入输出统一路径空间。

## 4.2 数据流
1. 用户执行 CLI 提交任务或定时器触发任务。
2. Master 校验输入输出路径与参数，创建 Job 记录。
3. Master 扫描目录并生成 Task，写入 `tasks` 表并投递至 Redis Stream。
4. Worker 从消费组拉取消息，执行 ffprobe + ffmpeg。
5. Worker 将结果写回数据库并 ACK 消息。
6. Master 汇总统计并更新 Job 最终状态。

## 5. 模块设计
## 5.1 Master 模块
- `job_service`：创建、查询、终止 Job。
- `scan_service`：递归扫描、格式过滤、生成 task 清单。
- `dispatch_service`：将任务投递至队列。
- `aggregation_service`：汇总成功/失败/跳过、更新吞吐与耗时。
- `recovery_service`：回收超时任务、重新分发。

## 5.2 Worker 模块
- `consumer`：读取队列消息，控制并发消费。
- `probe_executor`：调用 ffprobe，提取媒体元信息。
- `transcode_executor`：构建 FFmpeg 命令并执行。
- `result_reporter`：写入 task 结果与失败原因。
- `heartbeat`：周期上报 worker 存活状态。

## 5.3 公共模块
- `config`：加载 YAML + 环境变量 + CLI 覆盖。
- `path_mapper`：统一输入输出路径转换与层级保留。
- `security_guard`：命令参数白名单校验与安全转义。
- `metrics`：暴露 Prometheus 指标。

## 7. 状态模型设计
## 7.1 Job 状态机
- `PENDING`：任务已创建，待扫描/分发。
- `RUNNING`：已进入执行阶段。
- `PARTIAL_SUCCESS`：至少一个 Task 失败，至少一个 Task 成功。
- `SUCCESS`：所有 Task 成功。
- `FAILED`：全部失败或创建阶段失败。
- 迁移规则：
  - `PENDING -> RUNNING -> SUCCESS | PARTIAL_SUCCESS | FAILED`

## 7.2 Task 状态机
- `PENDING`：待执行。
- `DISPATCHED`：已投递队列。
- `RUNNING`：Worker 正在处理。
- `SUCCESS`：转码成功。
- `FAILED`：达到最大重试后失败。
- `RETRYING`：失败后待重试。
- `SKIPPED`：格式不支持或策略跳过。

## 9. 队列与消息协议
## 9.1 Stream 与消费组
- Stream：`transcode_tasks`
- Consumer Group：`transcode_workers`
- Consumer Name：`{worker_id}-{pid}`

## 9.2 Task 消息体
```json
{
  "task_id": "uuid",
  "job_id": "uuid",
  "input_path": "/data/in/a.mp4",
  "output_path": "/data/out/a_h265.mp4",
  "params": {
    "video_codec": "libx265",
    "crf": 28,
    "preset": "medium",
    "audio_codec": "aac"
  },
  "retry_count": 0,
  "max_retry": 2
}
```

## 9.3 ACK 与重投策略
- Worker 成功写库后执行 ACK。
- Worker 异常退出导致未 ACK 的消息由 Master 定时 reclaim。
- reclaim 阈值：`task_timeout_sec * 1.5`。

## 10. 转码引擎设计
## 10.1 探测命令（ffprobe）
```bash
ffprobe -v error -print_format json -show_format -show_streams "<input>"
```

## 10.2 MP4 输出命令模板
```bash
ffmpeg -y -i "<input>" \
  -c:v libx265 -preset medium -crf 28 -g 48 \
  -c:a aac -b:a 128k \
  "<output>.mp4"
```

## 10.3 M3U8 输入转 MP4 模板
```bash
ffmpeg -y -protocol_whitelist "file,http,https,tcp,tls,crypto" -i "<input>.m3u8" \
  -c:v libx265 -preset medium -crf 28 -g 48 \
  -c:a aac -b:a 128k \
  "<output>.mp4"
```

## 10.4 参数白名单
- 视频：`video_codec/crf/bitrate/fps/scale/gop/preset/tune`
- 音频：`audio_codec/audio_bitrate/audio_channels/audio_sample_rate`
- 容器：`mp4|ts`
- 非白名单参数拒绝执行，防止命令注入与误配。

## 11. 调度策略与并发控制
## 11.1 P1 调度策略
- 全局策略：队列拉取（天然削峰、实现简单）。
- Worker 侧：固定并发池（线程池/协程池），默认 `min(4, CPU核数/2)`。
- Master 侧：按扫描顺序投递任务，保障行为确定性。

## 11.2 任务超时
- 每个 Task 设置 `task_timeout_sec`。
- 超时后标记失败并进入重试逻辑。

## 11.3 重试策略
- 默认最多重试 2 次（总执行 3 次）。
- 指数退避：`5s, 15s`。
- 仅对可重试错误重试（网络抖动、临时 I/O 失败）；格式损坏不重试。

## 12. 存储与路径策略
## 12.1 输入扫描规则
- 递归扫描输入根目录。
- 扩展名初筛 + ffprobe 复核。
- 不支持文件标记为 `SKIPPED` 并记录原因。

## 12.2 输出路径规则
- 文件名后缀规则：`{basename}{suffix}.{ext}`，如 `a_h265.mp4`。
- 目录模式：
  - `flat`：全部输出到 output_root。
  - `mirror`：保持输入层级到输出目录。

## 12.3 原子写入策略
- 输出先写临时文件 `*.tmp`。
- 转码成功后原子 rename 到目标文件名，避免脏文件被误用。

## 13. CLI 设计
## 13.1 命令列表
- `submit`：提交 Job
- `status`：查询 Job 状态
- `retry-failed`：重试失败 Task
- `run-master`：启动 Master 服务
- `run-worker`：启动 Worker 服务
- `run-scheduler`：启动定时触发器（可合并到 master）

## 13.2 submit 示例
```bash
python -m app.cli.main submit \
  --input-root /data/in \
  --output-root /data/out \
  --suffix _h265 \
  --mode mirror \
  --crf 28 \
  --max-retry 2 \
  --concurrency 4
```

## 16. 容错与恢复设计
- Master 启动恢复：扫描 `RUNNING` Job，将卡住的 `RUNNING/DISPATCHED` Task 重新入队。
- Worker 失联判定：`last_heartbeat > 3 * heartbeat_interval`。
- 消息幂等：Task 执行前以 `task_id + status` 做幂等检查，避免重复转码。
- 输出幂等：目标文件存在时按策略：
  - `overwrite=true` 覆盖
  - `overwrite=false` 跳过并标记 `SKIPPED`

## 17. 安全设计
- 禁止将存储访问密钥写入日志。
- 外部命令必须使用参数数组调用，不使用 shell 拼接字符串。
- 输入路径需做目录越权校验（必须在允许根路径内）。
- 对 m3u8 外部 URL 可选白名单域名限制（P1 可配置开关）。

## 19. 测试与验证方案
## 19.1 单元测试
- 参数校验、路径映射、命令构建、状态机迁移、重试策略。

## 19.2 集成测试
- 单机：提交 Job -> 处理完成 -> 汇总正确。
- 多机：2~4 Worker 并行，验证吞吐提升与状态一致性。
- 异常：模拟 Worker 崩溃，验证任务回收与重分发。

## 19.3 验收对齐
- 对齐 P1 第 12 章验收标准，输出测试报告：
  - 成功率
  - 平均耗时
  - 失败分类
  - 重试命中率

## 20. 容量与性能估算
- 吞吐估算公式：
  - `总吞吐 ≈ Worker数 × 单Worker并发 × 单并发平均处理速率`
- 初始建议：
  - CPU 转码：每 Worker 并发 1~2
  - 磁盘/网络富余时可提升到 3~4
- 观察指标：
  - CPU 利用率 > 85% 且队列积压增长，优先扩 Worker
  - I/O 等待高，优先优化共享存储吞吐

## 21. 实施计划（技术落地）
- S1：搭建项目骨架与配置系统（CLI、日志、数据库、Redis）。
- S2：完成单机 Worker 转码闭环（ffprobe + ffmpeg + 结果写库）。
- S3：完成 Master 扫描、分发、汇总与状态机。
- S4：接入定时触发、重试、超时回收、心跳机制。
- S5：完成 Docker 化、多节点联调、监控指标与验收测试。

## 22. 风险与规避
- 共享存储性能瓶颈：压测后确定最佳并发并限制瞬时峰值。
- M3U8 外链不稳定：启用重试与超时策略，必要时预下载。
- FFmpeg 参数误配：白名单 + 参数校验 + 模板化配置。
- 数据库写入热点：批量更新与索引优化（`job_id/status/updated_at`）。

## 23. P2 演进兼容点
- 队列从 Redis Streams 平滑升级到 Kafka/RabbitMQ。
- Master 从单点升级为主备/选主。
- 引入 Web 控制台、权限、多租户与高级调度策略。
